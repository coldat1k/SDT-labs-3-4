## Інструкція із запуску автоматизації

### Розгортання інфраструктури (Terraform)

Для розгортання інфраструктури за допомогою Terraform необхідно послідовно виконати такі команди в корені проєкту:

```bash
terraform init
terraform plan -out=main.tfplan
terraform apply "main.tfplan"
```

### Управління конфігурацією (Ansible)

Для управління конфігурацією та налаштування сервісів за допомогою Ansible необхідно перейти у відповідну директорію та запустити плейбук:

```
cd ansible
ansible-playbook -i inventory.ini site.yml
```

> **Примітка:** у разі тестування конфігурацій на локальному контурі через апаратні обмеження вкладеної віртуалізації процесора на хості, запуск плейбука виконується на localhost із підвищенням привілеїв:

```
ansible-playbook -i inventory.ini site.yml -K
```

## Відповідність критеріям приймання роботи

### Автоматизація

Створення всієї базової інфраструктури, включаючи дві віртуальні машини на основі офіційного образу Ubuntu, хмари метаданих cloud-init та мережеві інтерфейси, повністю автоматизовано та виконується однією командою `terraform apply`. Подальше розгортання програмного забезпечення, системи керування базами даних, конфігурація користувачів та веб-сервера виконується за один запуск `ansible-playbook`.

### Ідемпотентність

Усі розроблені Ansible-ролі є повністю ідемпотентними. Повторне виконання команди `ansible-playbook` аналізує поточний стан операційної системи й не вносить жодних нових змін чи модифікацій у конфігураційні файли, пакети чи облікові записи користувачів, якщо система вже перебуває у цільовому описаному стані.

### Декларативність

У сценаріях конфігурації Ansible повністю відсутні імперативні системні виклики через модулі `shell` чи `command`. Налаштування середовища реалізовано виключно через нативні декларативні модулі:

- `apt` — для встановлення пакетів
- `user` — для управління користувачами
- `template` — для генерації конфігураційних файлів через Jinja2
- `systemd` — для фонових служб
- `ufw` — для брандмауера

### Розподіленість та ізоляція Бази Даних

Веб-застосунок `app.js` на першій машині успішно підключається до СУБД на другій машині за допомогою рядка підключення `DATABASE_URL`, що динамічно генерується Ansible у файлі середовища `.env`.

Прямий доступ до бази даних ззовні повністю заблоковано:

- **на рівні СКБД** (у файлі `pg_hba.conf` доступ дозволено лише для IP `192.168.122.150`)
- **на рівні ОС** (файрвол `ufw` на db-node відхиляє всі вхідні пакети на порт `5432`, окрім запитів із хоста worker)

## Керування користувачами та правами доступу

Сценаріями автоматизації на вузлах розгорнуто чотири ізольовані облікові записи:

| Користувач | Присутність | Призначення | Права |
|---|---|---|---|

| `teacher` | Всі ВМ | Перевірка виконання роботи | Пароль `12345678`, доступ за SSH, ескалація до root зі своїм паролем |

| `operator` | Тільки VM1 | Оперативне керування системою | Пароль `12345678`, доступ за SSH, обмежений `sudo` (без пароля) на: запуск/зупинку/reload/статус застосунку та reload nginx |

## Перевірка працездатності (Health Checks)

Маршрутизація діагностичних запитів налаштована через конфігураційні шаблони Nginx:

| Ендпоінт | Обробка | Призначення |
|---|---|---|
| `/health/alive` | Безпосередньо Nginx, повертає `200 OK` + `{"status": "alive"}` | Перевірка працездатності проксі-компонента |
| `/health/ready` | Проксується на Node.js застосунок. Виконує `SELECT 1` до PostgreSQL | Перевірка зв'язку з БД. При недоступності БД повертає `503 Service Unavailable` |

## Інструкція з розгортання проєкту на іншому комп'ютері

Якщо інфраструктура розгортається у середовищі, де успішно працює KVM/libvirt, перед запуском Ansible необхідно:

1. Виконати команду `terraform apply` для створення реальних віртуальних машин.
1. Отримати фактичні IP-адреси створених вузлів за допомогою `terraform output`.
1. Оновити файл `ansible/inventory.ini`, замінивши значення `localhost` на реальні отримані IP-адреси:

```
[workers]
worker-node ansible_host=<ФАКТИЧНИЙ_IP_VM1>

[db]
db-node ansible_host=<ФАКТИЧНИЙ_IP_VM2>

[all:vars]
ansible_user=ansible
ansible_ssh_private_key_file=~/.ssh/id_rsa
variant_number=23
```

1. Запустити налаштування конфігурації в бойовому режимі:

```
ansible-playbook -i inventory.ini site.yml
```
